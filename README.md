# Ozon Order Bot — заказы и этикетки в Telegram

Telegram-бот для продавца Ozon (схемы **FBS/rFBS**), умеет вести **несколько
магазинов одновременно** в один Telegram-чат. Опрашивает Ozon Seller API
(без вебхука, без своего публичного сервера) и:

1. При новом отправлении сразу присылает в Telegram номер, название товара
   и его главное фото.
2. Как только отправление переходит в статус «готово к отгрузке», бот сам
   получает этикетку, дописывает снизу артикул продавца (`offer_id`) и
   присылает её же сообщением.
3. По команде собирает **один PDF** сразу со всеми ещё не напечатанными
   этикетками — можно исключить часть или указать конкретные номера.

## Запуск

**Вариант 1:** двойной клик по **`Запуск.command`**.

**Вариант 2 (терминал):**
```bash
cd ozon-order-bot
./venv/bin/python bot.py
```

Бот работает в терминале, пока не остановите (`Ctrl+C`). При падении сам
перезапускается через 5 секунд.

## Настройка (один раз)

1. `cp .env.example .env`
2. Для **каждого** магазина: его кабинет Ozon → **Настройки → Seller API** →
   сгенерировать ключ, вписать в `.env` блоком `SHOP_N_NAME`/`SHOP_N_CLIENT_ID`/
   `SHOP_N_API_KEY` (нумерация с 1 подряд, без пропусков). Один магазин —
   удалите второй блок; больше двух — добавьте `SHOP_3_*` и т.д.
3. Токен бота у **@BotFather** в Telegram → `BOT_TOKEN` в `.env`.
4. `CHAT_ID` пока можно оставить пустым/произвольным, запустить бота,
   написать ему `/start` в личку — он ответит вашим настоящим `chat_id`.
   Впишите его в `.env` и перезапустите бота.

Оба (все) магазина шлют уведомления в один и тот же чат — каждое сообщение
подписано именем магазина (`SHOP_N_NAME`), чтобы не путать, откуда заказ.

Зависимости уже установлены в `venv/`. Если переносите проект на другой
компьютер: `python3 -m venv venv && ./venv/bin/pip install -r requirements.txt`.

## Кнопки бота

Под полем ввода в Telegram — постоянная клавиатура:

- **📋 Ожидают** — что ещё не напечатано
- **🖨 Печать** — выбор этикеток кнопками: по умолчанию отмечены все
  готовые, нажатием можно исключить лишние (или «⬜ Снять все» и отметить
  только нужные) — и собрать один PDF
- **🔁 Повтор печати** — то же, но по уже напечатанным (из истории), по
  умолчанию ничего не выбрано — отмечаете, что нужно перепечатать
- **📜 История** — последние напечатанные

На экране выбора этикеток (🖨 Печать / 🔁 Повтор печати) есть кнопка
**🔎 Найти по номеру** — бот попросит ввести номер отправления (или его
часть, или название товара), покажет только совпадения; «Отметить
все»/«Снять все» в этот момент действуют только на найденное, а «❌
Сбросить поиск» возвращает полный список без потери отметок.

Без кнопок, только текстом (для тех, кто предпочитает печатать номера):
`/merge_select RU-123 RU-456`, `/reprint RU-123 ...`, `/stats`, `/ping`.

## Гарантия «без дублей»

Каждое отправление хранится в БД одной строкой с nullable-полями
`notified_at`/`label_sent_at`/`printed_at` — как только соответствующее
действие один раз выполнено успешно, дальнейшие циклы поллинга просто
пропускают эту строку (`WHERE notified_at IS NULL` и т.п.), даже после
перезапуска бота: статус хранится в `bot_state.sqlite3`, а не в памяти.
Это верно независимо от того, из какого магазина отправление — `posting_number`
Ozon уникален по всей площадке, поэтому совпадение номеров между двумя
разными магазинами не приведёт к перепутыванию заказов.

## Важно: непроверенные вживую предположения об API Ozon

Часть деталей Ozon Seller API была неизвестна на момент написания кода —
публичная документация Ozon была недоступна для автоматической проверки.
**2026-07-07: бот впервые запущен вживую на реальных заказах** — список
отправлений, поля `products[]`, статус `cancelled`, а также главное фото
товара (`/v3/product/info/list`, поле `images[0]`) подтвердились и работают
как задумано. Заодно вскрылся и был исправлен один реальный баг: CDN Ozon
(`ir.ozone.ru`) блокирует запросы с серверов Telegram при отправке фото по
прямой ссылке (антибот) — `sendPhoto` теперь сам скачивает картинку и
загружает её файлом, а не передаёт ссылку.

Что ещё остаётся непроверенным вживую (батч этикеток на >1 отправление,
порядок страниц в объединённом PDF, форма ошибки при невалидном
posting_number — ещё не было случая с несколькими готовыми к печати
заказами одновременно) — отмечено комментарием `ПРОВЕРИТЬ` в
`ozon_client.py` и `pdf_label.py`.

**Прежде чем доверять боту в бою**, запустите один раз (с реальными ключами
и хотя бы одним реальным заказом):
```bash
./venv/bin/python scripts/verify_label_api_assumptions.py
```
Скрипт ничего не меняет и никуда не пишет — только печатает диагностику и
сохраняет пробные PDF в `scripts/output/` для проверки глазами. Он же
подскажет, если какое-то предположение (например, порядок страниц в
объединённой этикетке) не подтвердилось — код в `pdf_label.py` устроен так,
чтобы в таком случае безопасно откатываться на более медленные, но
гарантированно верные поштучные запросы, а не молча ошибаться.

## Состав

```
ozon-order-bot/
├── bot.py               # поллинг, Telegram, команды/кнопки, авто-рестарт
├── ozon_client.py        # обёртка над api-seller.ozon.ru
├── pdf_label.py           # наложение артикула, батчинг/бисекция, склейка PDF
├── db.py                   # состояние (aiosqlite)
├── config.py                # .env -> константы + список магазинов (SHOP_N_*)
├── assets/DejaVuSans-Bold.ttf  # шрифт с кириллицей для наложения на этикетку
├── scripts/verify_label_api_assumptions.py  # разовая проверка вживую
├── Запуск.command          # запуск двойным кликом (macOS)
└── .env                     # секреты (создать из .env.example, не в git)
```
